# stock-dashboard CODEBASE

## 1. 목적
이 문서는 stock-dashboard의 실제 코드베이스 운영 규칙을 정리한다.
ARCHITECTURE.md가 구조와 책임 분리를 설명한다면, 이 문서는 파일·함수·캐시·예외 처리·데이터 흐름의 실제 구현 관행을 기록한다.

## 2. 작업 원칙
- 가능한 한 로컬 DB를 우선 읽고, 부족한 부분만 API로 보충한다.
- 수집 로직과 분석 로직은 분리한다.
- 분석용 계산은 재현 가능해야 하므로 기준일과 원천 소스를 함께 남긴다.
- 마크다운 링크 자동 변환 아티팩트는 평문 경로로 정리한다.
- 경로 표기는 `core/db.py`처럼 평문으로 쓴다.

## 3. 주요 코드 영역
### core/
- DB 접근
- 원천 데이터 수집
- 전처리
- 공통 유틸리티

### stock-dashboard/
- 대시보드 실행 코드
- 분석 결과 렌더링
- 종목/포트폴리오 분석 진입점

### profiles/
- 종목별 보정 파일
- 예: `profiles/010170.md`

### prompts/
- 반복 실행용 프롬프트 템플릿
- 단일 종목, 포트폴리오, 계좌 단위 실행문

### docs/
- API 메모
- 버그 기록
- 계산 규칙
- 운영 체크리스트

## 4. DB 운영 규칙
- 가격/수급/뉴스/지수 데이터는 가능한 한 DB에 캐시한다.
- 동일 지표는 한 번 정한 기준으로만 저장한다.
- 결측은 NULL 또는 명시적 플래그로 관리한다.
- 단순 재조회보다 적재 우선이 원칙이다.
- 최신 거래일 갱신은 증분 방식으로 처리한다.

## 5. 자주 쓰는 데이터셋
- `prices`
- `investor_flow`
- `short_balance`
- `member_daily`
- 뉴스 캐시
- 리서치 캐시
- KOSPI/KOSDAQ OHLCV 캐시
- 이벤트 로그 캐시

## 6. 알려진 수집/분석 패턴
- 종목별 상승률 계산은 동일 기간의 시작일과 종료일을 기준으로 한다.
- 지수 대비 상대강도는 종목 OHLCV와 지수 OHLCV의 동일 구간 수익률로 계산한다.
- 거래량 급증 여부는 평균 거래량 대비 배수로 확인한다.
- 수급 해석 시 개인, 외국인, 기관을 동시에 본다.
- 공매도/대차는 분모 정의를 함께 기록한다.

## 7. 자주 발생하는 주의점
- signals_history가 비어 있을 수 있다.
- 일부 지수 시계열은 초기 적재가 필요하다.
- 뉴스 캐시는 타임스탬프 지연이 있을 수 있다.
- 공시와 뉴스는 시점 차이가 날 수 있다.
- 마크다운 링크가 평문 경로를 자동으로 감싸는 경우가 있으므로 저장 전에 확인한다.

## 8. 예외 처리
- 데이터가 부족해도 분석은 멈추지 않는다.
- 불완전한 데이터는 미측정 또는 보수적 판정으로 표시한다.
- 동일 이벤트를 여러 소스가 다르게 표현하면 공식성 높은 원문을 우선한다.
- 실시간성보다 재현성과 일관성이 중요할 때는 DB 값을 우선한다.

## 9. 단위 규칙
- 가격: 원.
- 거래량: 주식 수.
- 거래대금: 원.
- 수익률: 퍼센트.
- 기간: 거래일 기준과 달력 기준을 혼동하지 않는다.
- 상대강도: 동일 구간 비교를 원칙으로 한다.

## 10. 유지보수 규칙
- 새로운 모듈이나 데이터 소스를 추가하면 이 문서를 먼저 갱신한다.
- 큰 구조 변화는 ARCHITECTURE.md에, 세부 구현 관행은 CODEBASE.md에 둔다.
- 분석 규칙은 CLAUDE.md에만 둔다.
- 종목별 예외는 profiles/로 분리한다.

## 11. 작업 체크리스트
새 기능을 추가할 때:
1. 원천 데이터 확인
2. DB 적재 방식 정의
3. 재조회/캐시 정책 정의
4. 분석 파이프라인 영향 확인
5. 문서 갱신
