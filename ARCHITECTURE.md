# stock-dashboard ARCHITECTURE

## 1. 프로젝트 목적
이 프로젝트는 한국주식 분석 및 모니터링을 위한 데이터 수집·저장·분석 시스템이다.
주요 목표는 종목별 시계열, 수급, 공시, 뉴스, 지수 데이터를 통합해 분석 에이전트가 바로 활용할 수 있는 형태로 제공하는 것이다.

## 2. 핵심 역할 분리
- 수집 계층: KIS, KRX, Naver, DART 등 외부 소스에서 원천 데이터를 가져온다.
- 저장 계층: 로컬 DB에 적재하여 재조회 비용을 줄이고 분석 일관성을 높인다.
- 분석 계층: CLAUDE.md의 분석 규칙을 적용해 종목 해석, 이벤트 추출, 경고 판정을 수행한다.
- 보조 계층: 시장환경 지표, 상대강도, 파생 지표를 계산한다.

## 3. 디렉토리 개요
- `core/`
  - DB 접근, 수집 로직, 전처리, 공통 유틸리티가 위치한다.
- `stock-dashboard/`
  - 분석 관련 실행 파일, 대시보드, 에이전트 연동 코드가 위치한다.
- `profiles/`
  - 종목별 선택 보정 파일을 둔다. 예: `profiles/010170.md`
- `prompts/`
  - 재사용 가능한 실행 프롬프트를 둔다.
- `docs/`
  - API 사용법, 스키마, 작업 규칙, 알려진 이슈를 분리하여 기록한다.

## 4. 데이터 소스 역할
### KIS API
- 가격, 거래량, 수급, 공매도, 계좌성 시세 데이터의 우선 원천.
- 종목 OHLCV와 수급 관련 데이터의 최신성 확보에 사용한다.

### KRX Open API
- KOSPI, KOSDAQ, 업종 지수 및 시장 메타 데이터 확보에 사용한다.
- 시장환경 비교, 상대강도 계산, 지수 시계열 백필에 사용한다.

### Naver / 뉴스
- 뉴스 헤드라인, 보도 흐름, 시장 반응 보조 확인용.
- 공식 공시를 대체하지 않는다.

### DART / 공시
- 실적, 증자, 인수합병, 주요 계약, 정정 공시의 최우선 원문 소스.
- 해석보다 원문 확인을 우선한다.

## 5. 로컬 DB 운용 원칙
- 이미 정제된 데이터는 가능한 한 DB를 우선 사용한다.
- 지수 OHLCV, 종목 OHLCV, 수급, 공시 캐시를 축적해 재조회 비용을 줄인다.
- DB에 없는 기간만 원천 API로 보충 수집한다.
- 동일 지표는 출처를 섞지 말고 단일 기준으로 저장한다.
- 저장 시 기준일과 시각을 명시한다.

## 6. 시장환경 데이터
- KOSPI, KOSDAQ, 비교 업종 지수는 별도 시계열로 저장한다.
- 종목 분석 시 상대강도 계산을 위해 같은 기간의 지수 데이터를 유지한다.
- 시장환경 데이터가 없으면 분석은 진행하되, 상대강도는 미측정으로 표시한다.
- 시장환경 데이터는 종목 분석을 보조하는 입력값이며, 종목 고유 이벤트 해석을 대체하지 않는다.

## 7. 분석용 표준 데이터셋
분석에 자주 쓰는 핵심 테이블:
- `prices`
- `investor_flow`
- `short_balance`
- `member_daily`
- 뉴스/리서치 캐시
- 지수 OHLCV 캐시
- 이벤트 로그 캐시

## 8. 알려진 동작 규칙
- 1일 단위 캔들 데이터와 거래일 기준 정렬을 우선한다.
- 결측일이 있으면 휴장인지 수집 실패인지 구분한다.
- 급등/급락 분석 시 거래량 급증 여부를 같이 본다.
- 수급 해석은 개인/외국인/기관을 함께 본다.
- 공매도 수치는 분모 정의를 명시한다.

## 9. 단위 규칙
- 가격은 원 단위.
- 거래량은 주식 수.
- 거래대금은 원 단위.
- 수익률은 퍼센트.
- 기간 표기는 거래일 기준과 달력 기준을 혼동하지 않는다.
- 상대강도와 누적수익률은 동일 구간 기준으로 계산한다.

## 10. 개발 규칙
- 새로운 데이터 소스를 추가할 때는 적재 → 정제 → 저장 → 조회 순서를 문서화한다.
- 분석용 로직과 수집용 로직은 분리한다.
- 캐시가 있으면 먼저 조회하고, 부족분만 재수집한다.
- 경로를 마크다운 링크 자동변환 아티팩트로 쓰지 말고 평문 경로를 유지한다.

## 11. 알려진 이슈
- 일부 기간의 지수 OHLCV가 비어 있을 수 있다.
- signals_history가 비어 있는 경우가 있다.
- 뉴스 캐시는 시점별 중복과 지연 반영 가능성이 있다.
- 공시/뉴스/수급 간 타임스탬프가 다를 수 있다.
- 분석 결과는 원천 데이터의 최신성과 일치 여부를 항상 확인해야 한다.

## 12. 유지보수 원칙
- 구조 변경 시 ARCHITECTURE.md를 먼저 갱신한다.
- CLAUDE.md에는 운영 규칙만 남기고, 구조 설명은 이 문서에 둔다.
- 종목별 예외는 `profiles/`로 분리한다.
- 자주 바뀌는 세부 구현은 본 문서에 길게 쓰지 않는다.
